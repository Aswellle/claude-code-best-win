## 二、根因深度分析

### 第一轮修复的误判

上一轮使用了 `cmd /k ""D:\Git Clone\...\claude-core.exe""` 模式（double-double-quote），但该模式对 wt.exe 的参数解析器来说适得其反。

### 三层解析器的冲突

启动链涉及三个解析器，各自规则不同：

```
PowerShell Start-Process
    ↓ (传递 ArgumentList 字符串)
wt.exe (使用 CommandlineToArgvW / CRT 规则)
    ↓ (重新构造 cmd.exe 命令行)
cmd.exe /k "..." (自有引号剥离规则)
```

**wt.exe (CRT) 对 `""D:\Git Clone\...\exe""` 的解析**：
- 第一个 `"` → 进入 quoted 区域
- 第二个 `"` → 在 quoted 区域内遇到 `"` = 退出 quoted 区域（CRT 规则）
- 之后 `D:\Git Clone\...` 处于 **未 quoted** 状态
- 遇到空格 → token 在 `D:\Git` 处截断

结论：double-double-quote 修复使 wt.exe 把路径拆分，根本无效。

**cmd.exe `/k "path"` 的问题**：
- 即使 wt.exe 能正确传递，cmd.exe 的 `/k` 会剥离外层引号
- 剥离后路径无引号保护 → 按空格分割

这两个问题是结构性矛盾，无法通过调整引号层数解决。

---

## 三、解决方案：PowerShell -EncodedCommand

**核心思路**：彻底放弃 `cmd /k`，改用 PowerShell 的 `-EncodedCommand` 参数传递 base64 编码命令。

**为什么 base64 能解决问题**：
- base64 字符串只包含 `[A-Za-z0-9+/=]`，**无空格、无引号、无特殊字符**
- 所有中间解析器（wt.exe、PowerShell 参数处理）直接透传，不做任何分割
- PowerShell 解码后执行 `& 'path with spaces\exe' args`，原生支持含空格路径

**实际执行链**：

```powershell
# 1. 构造 PowerShell 命令（单引号路径，免疫空格）
$PsCommand = "& 'D:\Git Clone\CC-SRC\V5\packaging\output\claude-core.exe' --cwd C:\Users\welle"

# 2. Unicode base64 编码
$PsCmdEncoded = [Convert]::ToBase64String([Text.Encoding]::Unicode.GetBytes($PsCommand))
# → 纯 base64 字符串，无空格

# 3. 传给 wt.exe（没有任何需要特殊引号的内容）
wt.exe --title "Claude Code Best" powershell -NoLogo -NoExit -EncodedCommand <base64>
```

wt.exe 解析：`--title`, `"Claude Code Best"`, `powershell`, `-NoLogo`, `-NoExit`, `-EncodedCommand`, `<base64>` ← 每个 token 都是干净的，无歧义

PowerShell 执行：`& 'D:\Git Clone\...\claude-core.exe' --cwd C:\Users\welle` ← 单引号路径，PowerShell 原生处理空格 ✓
